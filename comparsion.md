## 七、基础设施事件与处置（诚实记录）

评测中途遭遇**真实故障并已修复**，记录如下以保证可复现与结果可信：

1. **aiberm 账户余额耗尽**：约 00:10（qwen 全量运行期间，叠加多模型并发的消费高峰），aiberm 主余额被打到负值（`用户额度不足, 剩余额度 ¥-0.06`）。
2. **故障是按模型/通道的**：经直连逐一验证，**只有 `qwen3.7-max` 这一个通道**返回 403 额度不足；`deepseek-v4-flash / kimi-k2.7-code / glm-5.2 / gemini-3.5-flash / gpt-5.4-mini(judge)` 全部仍 200 OK。故其余 4 模型与打分均不受影响、结果有效。
3. **后果**：qwen3.7-max 首跑 88/94 题因 403 空答 → 误分 **0.054**（已留档为 `runs/cmp_qwen37max_FAILED_aiberm.eval.json`）。
4. **修复**：备用端点 dubrify 同样"Insufficient quota"且无对应模型版本；最终用**有余额的 `wismodel-proxy`（atominnolab）端点重跑 qwen3.7-max**（同名模型路由到同一阿里底层，仅计费/路由代理不同，对比仍有效；judge 仍走 aiberm）。重跑 **94/94、0 空答、0 错误 → 0.7893**。
5. **未补跑的空答（有意保留）**：glm-5.2 的 item 87、kimi-k2.7-code 的 item 67/87 为**模型自身未作答**（error=None，模型跑完循环但没产出，item 87 是难决策题、kimi 在其上跑满 15 轮仍空）——属真实模型鲁棒性信号，**不补跑**（补跑等于给个别模型额外重试、对其它模型不公平）。这与 qwen 的"基础设施 403"性质不同，故区别处理。

## 八、关键发现

1. **qwen3.7-max 是这批模型里唯一能逼近 opus 的**（0.789 vs 0.816，差距进入 judge 噪声带），赢在高权重维度（事实/机理/决策/数值）的全面领先与稳定的作答纪律。
2. **glm / deepseek / kimi 三者打平在 0.75**，但风格迥异：glm 省 token 鲁棒（越难越强）、ds 广撒网耗 token、kimi 爱循环收敛弱。若看**性价比，glm-5.2 最优**（半数 token 拿到同档分数）。
3. **gemini-3.5-flash 的短板是"话少"**：中位 392 字的答案在"逐条写全事实/数值/范围+物理含义"的 v3 rubric 下系统性丢分；它 anti_hacking 最低反而暴露了"答得少"，而非"答得精"。
4. **系统的天花板由模型决定**：同一套 v11 系统，换上不同模型分数从 0.66 跨到 0.79——说明 DeepreadQA 把"检索召回 + 渐进阅读"的地基打好后，**最后一公里取决于模型的作答完整性与 agentic 纪律**，这正是 opus 强、gemini 弱的地方。

## 十、追加：intern-latest（2026-06-28）—— 能和名模型掰手腕吗？

**结论：暂时不能。** `intern-latest`（书生 intern-ai）在同一套 DeepreadQA v11 上拿 **0.6063**，是全榜最末——比 gemini-3.5-flash（0.660）还低约 0.05，距榜首 qwen3.7-max（0.789）约 0.18，距 opus 基线（0.816）约 0.21。它**能完整跑通 agentic 流程**（工具调用正常、7.1 轮/题、0 空答 0 错误），所以"接得进系统"没问题；差距在**答案质量**本身。

### 10.1 按 criterion 拆解（intern-latest vs 全场区间）

| criterion | intern-latest | 全场其余 5 模型区间 | 解读 |
|---|---|---|---|
| factual_anchor（主权重 ~56%） | **0.617** | 0.687–0.802 | 关键事实覆盖最弱，直接拖垮总分 |
| mechanism_explanation | 0.597 | 0.602–0.855 | 机理解释偏弱 |
| decision_logic | 0.662 | 0.690–0.831 | 决策逻辑偏弱 |
| numeric_precision | 0.522 | 0.609–0.826(qwen) | 数值精确偏弱 |
| process_completeness | **0.441** | 0.591–0.735 | 流程完整性全场最低 |
| comparative_balance | 0.667 | 0.500–0.833 | 中游 |
| anti_hacking（越低越好） | 0.112 | 0.031–0.174 | 中游，不靠"少说"取巧 |

- 短板是**全面性的**：除 anti_hacking/comparative 外，每个正向维度都处于或低于全场下沿，尤其 **factual_anchor（主权重）和 process_completeness** 明显落后。这不是某一类题偏弱，而是整体作答的事实密度与完整度不足。
- 题型：简答 0.625 / 主观 0.606 / 决策 0.562；难度：简单 0.739、中等 0.553、困难 0.591——**中等题塌方**（与 gemini 类似的形态）。
- 运行特征：迭代 7.1 轮、token 中位 11.3 万、答案中位 418 字、0 次触顶——行为正常，不是"没读够"或"循环爆掉"，纯粹是产出的答案质量不够。

### 10.2 工程说明（端点与限流，诚实记录）

- **端点**：intern-latest 走 **intern-ai 官方端点** `https://chat.intern-ai.org.cn/api/v1`（其余 5 模型走 aiberm；模型同名即同源，端点不同不影响分数）。
- **限流改串行**：首次按惯例用 8 分片并发，intern-ai 立刻限流——它以 **HTTP 400 + `code -20048「请求过于频繁」`** 返回（不是标准 429），导致 53 题里 44 题空答，该轮作废。改 **1 分片串行**重跑后全程 0 限流、0 空答、0 错误，得到上表干净结果。串行 94 题约 50 分钟。
- **工具调用**：intern-latest 支持 OpenAI 式 function calling（冒烟与全量均正常），是它能驱动 8 工具 agentic 循环的前提。
- 产物：`runs/cmp_internlatest.jsonl` / `.eval.json`；运行脚本在 scratchpad（含 intern key，**未入 git 仓库**）。

### 10.3 启示

intern-latest 作为 agent 的"可用性"达标（会用工具、不弃答、不崩），但在 CAE 这种**重噪声、中英混合、需把具体事实/数值写全**的任务上，作答质量与第一梯队（qwen/glm/ds/kimi）和 opus 还有明显代差。若要它掰手腕，杠杆不在系统而在模型本身的事实抽取与作答完整性——这与本报告对 gemini 的诊断一致。

## 十二、追加：VLM-OCR 语料修复（2026-07-04）—— 数据轴第一次真正破土

**动机**：0.816 之后 prompt 轴已 Pareto 饱和（§9/§11），失分账（94×6 逐题矩阵）显示
系统性低分题的首因是 **mineru 解析静默丢内容**：图/表/公式必丢（`![image](CDN)` 链接
已 403），中文双栏论文正文也整段丢（坝体论文 md 仅剩原文一半，HJC 参数段全无；
item 23 的 0.002 阈值、item 45 的内能吸收曲线均不在库里，模型只能诚实答"查不到"）。

**管线**（`deepread_sdk/pagediff.py` 纯函数 + `scripts/vlm_ocr_repair.py` 编排，
156 例测试全绿）：
1. **测算**：逐页"分段覆盖率"（按标点/控制符切段做包含匹配，抗嵌入式 OCR 错字——
   固定 shingle 会把 PhD 论文的假丢失误报成 80 页，段级度量修正为 19 页真丢）；
   选页 coverage<0.80→FULL 整页转写，否则有图→FIGURES 图表转写。8 篇 gold 共 286 页。
2. **转写**：gemini-3.5-flash 主（15/15 参数保真，与 opus-4.8 交叉验证曲线读数一致，
   且"贴坝面"类字形比 opus 更准）+ opus-4.8 兜底；逐页缓存可断点续跑。286 页 0 失败。
3. **回填**：逐行去重（已在 md 的行丢弃，误选页只浪费 token 不污染文档）→ 按"该页
   最后存活片段"锚点原位插入，块头带来源标记。
4. **重建**：复制 cae.db 作种子增量重建（content_hash 跳过 218 噪声篇，仅 8 篇 gold
   重富集，同一富集器保一致性）。

**三次迭代的机制课**（评测：opus-4.8 + 5 工具默认面，94 题 × 3 轮，v3 判分）：

| 版本 | 布局策略 | 3轮 mean_score | anchored | 关键逐题 |
|---|---|---|---|---|
| 基线 abl5 | 未修复 | 0.8139 (0.8160/0.8063/0.8193) | 0.8204 | item23=0.15 item45=0.32 item46=0.15 |
| v1 (vlm) | 转写原样内联 | **0.8282** (0.8208/0.8268/0.8370) | 0.8352 | 23→0.84 45→0.76 46→0.64；**但** 9/91/74/57 回归 −0.2~−0.4 |
| v2 (vlm2) | 标题降级内联 | 0.8208 (0.7970/0.8295/0.8359) | 0.8279 | 回归修回 4/6（9→0.67 91→0.74 57→0.87）；**但** 46 三轮全 0 |

- **v1 的病**：转写自带 `#` 标题把宿主小节劈碎（Benson 84→230 节），检索被导向
  "公式与参数取值"这类无上下文碎片节——毛收益（低分组 +0.14）被回归吃掉四成。
- **v2 的病**：标题降级治好碎片化（Benson 回到 114 节，多出的 32 节全是转写找回的
  **真实**子章节如"2.3.1 Contact algorithms"），但图表内容沉入大宿主节后**不可发现**：
  item 46 的 agent 搜到主题相近的"2.3.3 Damping"却再找不到侧壁缝隙那张图的转写。
- **合题（v3，vlm3）**：FULL 补录留在宿主节内（v2 行为），FIGURES 转写改放文档末尾
  **附录**，每页一个命名小节（`### 图表转写 · 原文第N页`，Benson 73 个）——结构不碎
  且每张图可检索、读取便宜。

**v3 结果与终局判定**：vlm3 三轮 0.8236/0.8173/0.8147，均值 0.8185（anchored 0.8258）。
item 45 三版最佳（0.806）、30 恢复（0.793），但 46 未获救（0.116）——三轮轨迹一致显示
其查询词（"位移衰减/阻尼/附加质量"）与证据关键词（"侧壁缝隙/对称边界"）天然错位，
证据可检索但查询到不了，属于查询改写/答案侧核验的范畴，布局救不了。

**判定**：三种布局总分 0.8185~0.8282（差 ≤0.01），小于版本内单轮波动（vlm2 三轮
0.797↔0.836）——布局轴已打到判分噪声底，就此收手。**跨布局稳健**的结论：
- 修复后总分全部 ≥ 基线：mean_score +0.005~+0.014，anchored +0.005~+0.015（3 轮均值）；
- 图表/数字硬上限**永久解除**：item23 0.15→0.84~0.94、item45 0.32→0.69~0.81（三版皆大幅）；
- 基线低分组（18 题）+0.09~0.14；高分组一致付出 −0.013~−0.020 的扰动代价。

默认库取 **v3 布局**（`store/cae_vlmocr.db`，`DEEPREAD_DB` 指向即用）：宿主节完整 +
图表按页命名可检索 + 找回 32 个真实子章节，结构最健全。v1 的更高点估计（0.8282）
源于碎片节的不可复现幸运检索，不作为生产形态。转写缓存在 `runs/vlm_ocr_cache/`
（286 页，重排布局零转写成本）。0.85 剩余缺口 ≈0.02~0.03，下一杠杆是答案侧
"核验-修补"回路（方案 axis ②），首批靶点即本轮暴露的 46/74/9/90 类
"读到但没说全 / 查询词错位"失分。

## 十三、追加：axis ② compose 核验-修补回路（2026-07-04）—— 中性收场，但机制课清晰

**设计**：compose 出稿后加一次审校（覆盖/数值显式/无依据论断 + 最多 2 条领域术语
检索探针），探针经 ToolBox 执行，再修一稿。`deepreadqa/verify.py`（协议解析+探针，
纯函数）+ `Config.verify_loop`（默认关，`DEEPREAD_VERIFY=1` 开）+ 13 项新测试。
PASS 短路不花修补钱；任何环节失败回退 composed——回路只能加分不能丢答案。

**两个变体，同库（cae_vlmocr.db v3）对照 vlm3 基线 0.8185**：

| 变体 | 修补动作 | 结果 (mean_score) | 逐题证据 |
|---|---|---|---|
| ver（重写版） | 整段改写 | 三轮 0.8049/0.7925/0.8095 = **0.8023（−0.016，有害）** | 高分题被重写毁掉：item47 满分 1.000→0.556、73 0.96→0.51；26 降 vs 15 升 |
| ver2（只增版） | 模型只产出补充要点，代码追加（物理上无法删改原文） | 单轮 **0.8173（≈基线，中性）** | 47 保住满分、49 +0.31、90 +0.17；但 8 题仍大跌 |

**为什么"只增"也到不了正收益**：判分日志直接给出了机制——item 9 的 rubric 带
**负向规则**（"不要给出 W^{1/3}/p_h^{-5/6} 形式"），审校判"缺经验关系式"→ 补充
要点把公式追加上去 → 负向规则触发扣分。这与 v12 的教训同构：在这套 v3 rubric
（负向准则 + anti_hacking）下，**补全度与踩坑率是 rubric 级耦合的**，答案侧的
"多说一点"天然双刃。旁证：增益题（49/90/48）多为探针带回新证据的真救援——
机制本身有效，但被负向准则的对称伤害抵消。

**判定**：axis ② 在本评测下中性收场，`verify_loop` 默认保持关闭；代码保留——
在没有负向准则的真实业务问答里，"探针补缺 + 只增不改"仍是低风险的质量杠杆。
0.85 的进一步讨论见下：注意本 rubric 下**参考答案自身仅得 ~0.814**（早前测定），
当前 v1 库 0.8282 已超参考答案；把 mean_score 推到 0.85 意味着要比参考答案高
0.036，其性质更接近拟合判分器的宽容结构而非知识质量提升。

> **结项（2026-07-04）**：经确认，正式把 **~0.83（3 轮均值）定为本评测口径（94 题 · v3 rubric · gpt-5.4-mini judge）下的诚实高原**，0.85 目标弃追（属"超参考答案 0.036"区间）。生产配置：5 工具默认面 + `store/cae_vlmocr.db`（v3 附录布局）+ compose 头 + `verify_loop` 关闭。四条轴（prompt/数据/答案/模型）的封顶证据分别见 §9、§12、§13 与 §10-11。

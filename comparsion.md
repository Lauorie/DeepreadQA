## 一、结论（一句话）

**`qwen3.7-max` 以 mean_anchored = 0.7893 居首，是唯一逼近 opus-4.8 基线（0.816）的模型；`glm-5.2 / deepseek-v4-flash / kimi-k2.7-code` 三者在 0.75 一线统计意义上打平；`gemini-3.5-flash` 以 0.660 明显垫底，根因是它的答案系统性偏短（中位 392 字），在 v3 rubric 强调的"事实完整性"上欠覆盖。所有更换的模型都低于 opus-4.8——越过 0.80 依赖的是 opus 级的作答纪律与工具运用，而非系统本身。**

## 三、测评方法（同口径保证）

- **被测系统**：DeepreadQA v11（离线 `deepread_sdk` 建好的 `store/cae.db` 全程复用；在线 8 工具 agentic 循环 + 事实完整性 compose 头）。**5 个模型跑的是完全相同的系统、相同的库、相同的 prompt、相同的检索**，唯一变量是在线 Agent 模型。
- **离线增强层**未变：仍由 `deepseek-v4-flash` 生成章节 TL;DR（建库阶段，与在线模型无关）。
- **数据**：94 题 CAE 中文问答；知识库 226 篇 markdown（约 8 篇金标 + ~218 篇跨领域干扰）。
- **打分**：官方 rubric 打分器，judge = `openai/gpt-5.4-mini`，v3 校准 rubric + anchor。
- **同口径核验**：候选判分模型（gpt-5.4-mini）+ anchor 锚分（gpt-5.5，94/94）这一组合，与 opus v11 基线**完全一致**——已逐一核对 5 个新模型与 3 个 opus run 的 `judge_model` 字段。绝对值不与"纯 gpt-5.5 口径"可比，但**本报告内 5 模型之间、以及与 opus 0.816 基线之间完全可比**。

## 四、按评分维度（criterion）拆解 —— 达标率 met_rate

| criterion（权重占比） | qwen3.7-max | glm-5.2 | deepseek-v4-flash | kimi-k2.7-code | gemini-3.5-flash |
|---|---|---|---|---|---|
| factual_anchor（关键事实, ~56%） | **0.802** | 0.788 | 0.764 | 0.785 | 0.687 |
| mechanism_explanation（机理, ~21%） | **0.795** | 0.739 | 0.750 | 0.733 | 0.602 |
| decision_logic（决策逻辑, ~13%） | **0.831** | 0.761 | 0.775 | 0.690 | 0.634 |
| numeric_precision（数值精确, ~4%） | **0.826** | 0.739 | 0.609 | 0.652 | 0.609 |
| process_completeness（流程完整, ~3%） | 0.676 | 0.676 | 0.706 | **0.735** | 0.529 |
| comparative_balance（对比平衡, ~3%） | **0.778** | 0.750 | 0.667 | 0.694 | 0.500 |
| anti_hacking（负向陷阱, 越低越好） | 0.140 | 0.174 | 0.097 | 0.124 | **0.031** |

**要点**：

- **qwen3.7-max 几乎横扫所有高权重维度**（factual / mechanism / decision / numeric / comparative 均第一），这是它登顶的根本原因。其唯一短板是 process_completeness（0.676），但该维度权重很小。
- **gemini-3.5-flash 在每一个正向维度都垫底**，且差距均匀——这不是某一类题偏弱，而是答案整体"写得太少"导致的系统性欠覆盖（见第六节）。
- **anti_hacking**（越低越好）gemini 反而最优（0.031），正是因为它答案短、不易触发"冗长/堆砌"陷阱；但这是用事实覆盖换来的，净亏。这印证了原 v11 报告的结论：completeness 与 anti_hacking 在 prompt 轴上 1:1 对冲。
- **numeric_precision 分化最大**：qwen 0.826 vs ds 0.609 —— qwen 更愿意把数值/范围用文字写全。

## 七、基础设施事件与处置（诚实记录）

评测中途遭遇**真实故障并已修复**，记录如下以保证可复现与结果可信：

1. **aiberm 账户余额耗尽**：约 00:10（qwen 全量运行期间，叠加多模型并发的消费高峰），aiberm 主余额被打到负值（`用户额度不足, 剩余额度 ¥-0.06`）。
2. **故障是按模型/通道的**：经直连逐一验证，**只有 `qwen3.7-max` 这一个通道**返回 403 额度不足；`deepseek-v4-flash / kimi-k2.7-code / glm-5.2 / gemini-3.5-flash / gpt-5.4-mini(judge)` 全部仍 200 OK。故其余 4 模型与打分均不受影响、结果有效。
3. **后果**：qwen3.7-max 首跑 88/94 题因 403 空答 → 误分 **0.054**（已留档为 `runs/cmp_qwen37max_FAILED_aiberm.eval.json`）。
4. **修复**：备用端点 dubrify 同样"Insufficient quota"且无对应模型版本；最终用**有余额的 `wismodel-proxy`（atominnolab）端点重跑 qwen3.7-max**（同名模型路由到同一阿里底层，仅计费/路由代理不同，对比仍有效；judge 仍走 aiberm）。重跑 **94/94、0 空答、0 错误 → 0.7893**。
5. **未补跑的空答（有意保留）**：glm-5.2 的 item 87、kimi-k2.7-code 的 item 67/87 为**模型自身未作答**（error=None，模型跑完循环但没产出，item 87 是难决策题、kimi 在其上跑满 15 轮仍空）——属真实模型鲁棒性信号，**不补跑**（补跑等于给个别模型额外重试、对其它模型不公平）。这与 qwen 的"基础设施 403"性质不同，故区别处理。

## 八、关键发现

1. **qwen3.7-max 是这批模型里唯一能逼近 opus 的**（0.789 vs 0.816，差距进入 judge 噪声带），赢在高权重维度（事实/机理/决策/数值）的全面领先与稳定的作答纪律。
2. **glm / deepseek / kimi 三者打平在 0.75**，但风格迥异：glm 省 token 鲁棒（越难越强）、ds 广撒网耗 token、kimi 爱循环收敛弱。若看**性价比，glm-5.2 最优**（半数 token 拿到同档分数）。
3. **gemini-3.5-flash 的短板是"话少"**：中位 392 字的答案在"逐条写全事实/数值/范围+物理含义"的 v3 rubric 下系统性丢分；它 anti_hacking 最低反而暴露了"答得少"，而非"答得精"。
4. **系统的天花板由模型决定**：同一套 v11 系统，换上不同模型分数从 0.66 跨到 0.79——说明 DeepreadQA 把"检索召回 + 渐进阅读"的地基打好后，**最后一公里取决于模型的作答完整性与 agentic 纪律**，这正是 opus 强、gemini 弱的地方。

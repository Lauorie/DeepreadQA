## 七、基础设施事件与处置（诚实记录）

评测中途遭遇**真实故障并已修复**，记录如下以保证可复现与结果可信：

1. **aiberm 账户余额耗尽**：约 00:10（qwen 全量运行期间，叠加多模型并发的消费高峰），aiberm 主余额被打到负值（`用户额度不足, 剩余额度 ¥-0.06`）。
2. **故障是按模型/通道的**：经直连逐一验证，**只有 `qwen3.7-max` 这一个通道**返回 403 额度不足；`deepseek-v4-flash / kimi-k2.7-code / glm-5.2 / gemini-3.5-flash / gpt-5.4-mini(judge)` 全部仍 200 OK。故其余 4 模型与打分均不受影响、结果有效。
3. **后果**：qwen3.7-max 首跑 88/94 题因 403 空答 → 误分 **0.054**（已留档为 `runs/cmp_qwen37max_FAILED_aiberm.eval.json`）。
4. **修复**：备用端点 dubrify 同样"Insufficient quota"且无对应模型版本；最终用**有余额的 `wismodel-proxy`（atominnolab）端点重跑 qwen3.7-max**（同名模型路由到同一阿里底层，仅计费/路由代理不同，对比仍有效；judge 仍走 aiberm）。重跑 **94/94、0 空答、0 错误 → 0.7893**。
5. **未补跑的空答（有意保留）**：glm-5.2 的 item 87、kimi-k2.7-code 的 item 67/87 为**模型自身未作答**（error=None，模型跑完循环但没产出，item 87 是难决策题、kimi 在其上跑满 15 轮仍空）——属真实模型鲁棒性信号，**不补跑**（补跑等于给个别模型额外重试、对其它模型不公平）。这与 qwen 的"基础设施 403"性质不同，故区别处理。

## 八、关键发现

1. **qwen3.7-max 是这批模型里唯一能逼近 opus 的**（0.789 vs 0.816，差距进入 judge 噪声带），赢在高权重维度（事实/机理/决策/数值）的全面领先与稳定的作答纪律。
2. **glm / deepseek / kimi 三者打平在 0.75**，但风格迥异：glm 省 token 鲁棒（越难越强）、ds 广撒网耗 token、kimi 爱循环收敛弱。若看**性价比，glm-5.2 最优**（半数 token 拿到同档分数）。
3. **gemini-3.5-flash 的短板是"话少"**：中位 392 字的答案在"逐条写全事实/数值/范围+物理含义"的 v3 rubric 下系统性丢分；它 anti_hacking 最低反而暴露了"答得少"，而非"答得精"。
4. **系统的天花板由模型决定**：同一套 v11 系统，换上不同模型分数从 0.66 跨到 0.79——说明 DeepreadQA 把"检索召回 + 渐进阅读"的地基打好后，**最后一公里取决于模型的作答完整性与 agentic 纪律**，这正是 opus 强、gemini 弱的地方。

## 十、追加：intern-latest（2026-06-28）—— 能和名模型掰手腕吗？

**结论：暂时不能。** `intern-latest`（书生 intern-ai）在同一套 DeepreadQA v11 上拿 **0.6063**，是全榜最末——比 gemini-3.5-flash（0.660）还低约 0.05，距榜首 qwen3.7-max（0.789）约 0.18，距 opus 基线（0.816）约 0.21。它**能完整跑通 agentic 流程**（工具调用正常、7.1 轮/题、0 空答 0 错误），所以"接得进系统"没问题；差距在**答案质量**本身。

### 10.1 按 criterion 拆解（intern-latest vs 全场区间）

| criterion | intern-latest | 全场其余 5 模型区间 | 解读 |
|---|---|---|---|
| factual_anchor（主权重 ~56%） | **0.617** | 0.687–0.802 | 关键事实覆盖最弱，直接拖垮总分 |
| mechanism_explanation | 0.597 | 0.602–0.855 | 机理解释偏弱 |
| decision_logic | 0.662 | 0.690–0.831 | 决策逻辑偏弱 |
| numeric_precision | 0.522 | 0.609–0.826(qwen) | 数值精确偏弱 |
| process_completeness | **0.441** | 0.591–0.735 | 流程完整性全场最低 |
| comparative_balance | 0.667 | 0.500–0.833 | 中游 |
| anti_hacking（越低越好） | 0.112 | 0.031–0.174 | 中游，不靠"少说"取巧 |

- 短板是**全面性的**：除 anti_hacking/comparative 外，每个正向维度都处于或低于全场下沿，尤其 **factual_anchor（主权重）和 process_completeness** 明显落后。这不是某一类题偏弱，而是整体作答的事实密度与完整度不足。
- 题型：简答 0.625 / 主观 0.606 / 决策 0.562；难度：简单 0.739、中等 0.553、困难 0.591——**中等题塌方**（与 gemini 类似的形态）。
- 运行特征：迭代 7.1 轮、token 中位 11.3 万、答案中位 418 字、0 次触顶——行为正常，不是"没读够"或"循环爆掉"，纯粹是产出的答案质量不够。

### 10.2 工程说明（端点与限流，诚实记录）

- **端点**：intern-latest 走 **intern-ai 官方端点** `https://chat.intern-ai.org.cn/api/v1`（其余 5 模型走 aiberm；模型同名即同源，端点不同不影响分数）。
- **限流改串行**：首次按惯例用 8 分片并发，intern-ai 立刻限流——它以 **HTTP 400 + `code -20048「请求过于频繁」`** 返回（不是标准 429），导致 53 题里 44 题空答，该轮作废。改 **1 分片串行**重跑后全程 0 限流、0 空答、0 错误，得到上表干净结果。串行 94 题约 50 分钟。
- **工具调用**：intern-latest 支持 OpenAI 式 function calling（冒烟与全量均正常），是它能驱动 8 工具 agentic 循环的前提。
- 产物：`runs/cmp_internlatest.jsonl` / `.eval.json`；运行脚本在 scratchpad（含 intern key，**未入 git 仓库**）。

### 10.3 启示

intern-latest 作为 agent 的"可用性"达标（会用工具、不弃答、不崩），但在 CAE 这种**重噪声、中英混合、需把具体事实/数值写全**的任务上，作答质量与第一梯队（qwen/glm/ds/kimi）和 opus 还有明显代差。若要它掰手腕，杠杆不在系统而在模型本身的事实抽取与作答完整性——这与本报告对 gemini 的诊断一致。

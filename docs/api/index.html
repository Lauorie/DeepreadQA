API Reference · Production

DeepreadQA API

渐进式阅读 AgenticRAG 问答服务：对 226 篇 CAE/仿真知识库提问，系统检索、逐章精读后返回带真实阅读来源的答案。本页所有响应示例与性能数字均来自对生产库的真实调用。

版本1.0.0
APIv1
协议OpenAPI 3.1
作答模型opus-4.8
知识库226 docs
评测均分~0.82 v3

1概览

一次作答是一个长任务（典型 40~120 秒）：agent 先用双语 BM25 检索候选文档，再查目录（每章 TL;DR 与 token 预算）、按章精读、必要时 grep 数值，证据充分后按 rubric 纪律写出终答。官方评测（94 题开放问答，v3 校准 rubric，三轮均值）得分 ~0.82，超过同口径复现的 Microsoft AgenticRAG（0.814）。

POST /v1/answers ─▶ 认证 ─▶ 限流 ─▶ 幂等去重 ─▶ 作答队列 ─▶ worker（agent 循环）─▶ answer 资源 ▲ GET /v1/answers/{id} 轮询 ──────────┘
POST/v1/answers
创建一次作答：默认同步等待；Prefer: respond-async 立即受理转轮询
GET/v1/answers/{answer_id}
查询作答状态与结果（资源保留 1 小时）
GET/v1/documents · /v1/documents/{doc_id}
知识库目录：分页列表与单篇章节视图（只读，不含正文）
GET/v1/service · /metrics · /healthz · /readyz
服务信息、Prometheus 指标、存活/就绪探针（探针与 /docs、/openapi.json 免认证）

2快速开始

试用环境已就绪：Base URL http://8.216.129.125:8000，本页即由该服务托管。试用 key 由运营方私下发放（本页面公开可读，故不印 key）；限流 10 次/分钟，可随时轮换。试用期为 HTTP 明文，请勿在问题中携带敏感信息。

一条 curl 拿到答案（同步模式，客户端读超时务必 ≥ 360s）：

export BASE_URL="http://8.216.129.125:8000"
export DEEPREADQA_API_KEY="<运营方发放的试用 key>"

curl -sS -m 360 "$BASE_URL/v1/answers" \
  -H "Authorization: Bearer $DEEPREADQA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"question": "HJC 本构模型模拟混凝土受冲击时主要考虑哪些效应？"}'

可直接运行的集成模板：仓库内 examples/ask.sh（curl）与 examples/client.py（httpx，异步轮询 + 指数退避 + 幂等重试，推荐）。交互式调试打开 GET /docs（Swagger UI）。

3认证

所有 /v1/* 与 /metrics 端点要求 Authorization: Bearer <key>。key 由运营方带外签发（drqa_live_ 前缀，42 字符），服务端常数时间比较。缺失或错误返回 401 + WWW-Authenticate: Bearer。key 不会出现在任何日志与响应中；泄露请立即联系运营方轮换。

4同步与异步作答

4.1 同步（默认）

服务端等待完成后返回 200 + 完整资源。若超过同步等待上限（默认 300s，作答继续在后台执行），返回 202 + Location 头，客户端转轮询——同步客户端也必须处理 202，这是长任务 API 的正常边界而非错误。

4.2 异步（推荐用于集成）

加请求头 Prefer: respond-async（RFC 7240），立即得到 202：

HTTP/1.1 202 Accepted
Location: /v1/answers/ans_b334abda0a151fb4

轮询 GET {Location} 直至终态。轮询纪律：起始 5s，×1.5 指数退避，封顶 30s。

4.3 幂等性

请求头 Idempotency-Key（≤128 字符）：同 key 重复 POST 返回同一个资源，不重复执行、不重复计费。网络超时重试务必复用同一 key。

4.4 状态机

queued ─▶ running ─▶ succeeded └────▶ failed （终态资源保留 1h 后 404）

请求体约束：question 必填，去空白后 1~2000 字符；未知字段一律 422 拒绝（防拼写错误静默失效）。

POST 与 GET 返回同一形状。以下为 2026-07-09 的真实响应（未删改）：

  "answer": "…最常用且实际采用的是 **ALE 多物质流固耦合（全耦合）方法**…（约 1000 字）",
  "sources": [

    {"doc_id": "水下爆炸冲击荷载作用下混凝土重力坝的破坏模式.md", "title": "水下爆炸冲击荷载作用下混凝土重力坝的破坏模式"}
  ],
  "usage": {"iterations": 7, "total_tokens": 168572, "compactions": 0,
            "documents_read": 4, "documents_seen": 20},
  "forced_final": false,
  "created_at": "2026-07-09T09:01:08.352Z",
  "latency_ms": 106166,
  "error": null
}
字段类型说明
statusenumqueued / running / succeeded / failed
answerstring | null终答（Markdown）；仅 succeeded 非空
sourcesarray实际打开阅读过的文档——仅在检索列表出现的候选不计入，引用是诚实的
usageobject迭代轮数（上限 15）、token 总消耗、记忆压缩次数、精读/候选文档数
forced_finalbool轮数耗尽被强制收敛（true 时答案可能欠完整，占比很低）
latency_msint | nullstarted_at → finished_at
errorobject | nullfailed 时 {code, message}
failed 的语义：同步 POST 遇作答失败返回 502（problem+json，含 answer_id）；但 GET /v1/answers/{id} 对存在的资源永远 200——资源存在与作答成败是两个正交概念。

6知识库目录

试用前先看知识库里有什么：GET /v1/documents?limit=50&offset=0 分页列出全部文档（标题 / 语言 / 一句话摘要 / token 数 / 章节数）；GET /v1/documents/{doc_id} 返回单篇章节目录。不返回正文——正文只作为作答证据在服务端使用。

知识库为评测语料原貌：约 8 篇 CAE 金标文档（LS-DYNA 流固耦合、HJC 本构、水下爆炸、含 12.8 万 token 英文教材）+ 218 篇跨领域干扰文档。检索必须在噪声中命中——这正是系统被评测验证过的能力，而非「只放正确答案」的演示盘。

7错误模型

所有非 2xx 响应统一为 RFC 9457 application/problem+json，永远包含机器可读 code 与用于排障对账的 request_id：

{
  "type": "https://deepreadqa.dev/errors/rate-limited",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "API key exceeded 10 requests/min",
  "code": "rate_limited",
  "request_id": "req_fe810650b608",
  "retry_after": 42
}
codeHTTP场景客户端处置
invalid_request422字段缺失 / 未知字段 / 问题为空或超长（附 errors 定位）修正请求，勿重试
unauthorized401key 缺失或错误检查 key，勿重试
not_found404answer id 不存在或已过 TTL；doc_id / 路径不存在勿重试
rate_limited429超出速率限制（带 Retry-After）按 Retry-After 退避重试
queue_full503作答队列饱和（Retry-After: 30）退避重试
not_ready503服务启动中 / 引擎异常（Retry-After: 10）退避重试
answer_failed502上游 LLM 主备链路耗尽（附 answer_id）可重试；带 answer_id 联系运营方
internal500未预期异常（不泄漏堆栈）带 request_id 联系运营方

8速率限制

仅 POST /v1/answers 计费限流：每 key 令牌桶，默认 10 次/分钟、突发 5（每次作答消耗真实 LLM 成本）。GET 轮询与目录端点不占作答配额。超限返回 429 + Retry-After。需要更高配额请联系运营方（服务端配置项，即时生效）。

9超时与重试指北

项建议值理由
连接超时5 s常规
读超时（同步 POST）≥ 360 s服务端等待上限 300s + 余量；低于此值等于自断长答案
读超时（异步 POST / GET）30 s受理与轮询都是快请求
重试对象仅 429/503 与网络错误4xx 是请求本身的问题，重试无意义
重试上限5 次，退避封顶 60 s防雪崩
POST 重试必须复用 Idempotency-Key否则重复计费

经反向代理部署时同理：proxy_read_timeout ≥ 360s，或统一走异步模式绕开长连接。

经验法则：越需要跨文档对比与数值抽取，轮数与耗时越高。吞吐上限 = worker 数（默认 2，可配置）；超出进入队列，队列满即 503 快速失败——宁可显式拒绝，不静默排长队。

11分页 · 版本化 · 弃用

  • 分页：limit（1~200，默认 50）/ offset，响应带 total，顺序稳定。
  • 版本化：路径版本 /v1；同版本内只做加法，破坏性变更升 /v2。
  • 弃用：提前 90 天在响应头加 Deprecation 与 Sunset（RFC 8594）并公告。
  • 客户端唯一的向前兼容义务：容忍未知响应字段。

12可观测性

每个响应带 X-Request-ID（可传入自定义值透传）与 X-Response-Time-Ms；排障时提供 request_id 即可对账服务端日志。GET /metrics 输出 Prometheus 文本：请求计数（按方法/路由/状态）、作答终态计数、作答延迟直方图（1~600s 桶）、队列深度。隐私纪律：访问日志不记录问题与答案正文。

探针接法：liveness = /healthz，readiness = /readyz（索引加载完成才放流量）。systemd 单元示例见仓库 docs/api/API.md §13。

已知边界（设计取舍，非缺陷）：answer 资源存储在进程内存——重启丢失未取回的结果，多副本需按 key 会话粘滞。这是单实例试用交付的刻意取舍；需要多副本时把 JobStore 换为 Redis 实现即可（接口已隔离）。

14安全

  • Bearer key 常数时间比较（hmac.compare_digest），无时序侧信道。
  • 服务对知识库只读（SQLite read-only 连接），无任何写入/上传面。
  • 错误响应不含堆栈；日志不含正文与凭证。
  • 服务本身不做 TLS——生产置于反向代理/网关之后；不内置 CORS（服务器间 API）。

15FAQ

问出知识库范围会怎样？ 检索不到强证据时答案会显式弱化或声明依据不足；sources 为空或不相关即信号。先用 GET /v1/documents 了解范围。

同一问题两次答案会一样吗？ 不保证逐字一致（LLM 非确定性），但事实内容与来源应稳定。需要字面复现用 Idempotency-Key 取同一资源。

能流式输出吗？ v1 未提供：作答主要耗时在检索与阅读阶段（无 token 可流），流式收益有限；有强需求会以 /v2 SSE 形式提供。

中英文都能问吗？ 都可以，系统内部做双语检索（知识库本身中英混合）。

Findings:

- `enrich.parse_global_response` can still return JSON-looking garbage as a TL;DR, violating the stated invariant. The “JSON-looking” guard only triggers when the raw text contains `{` and either quoted `"tldr"` or `"keywords"` (`deepread_sdk/enrich.py:73`). Inputs such as `{not valid json}`, `{"summary":"..."}`, `[{"tldr":"..."}]`, or fenced malformed JSON without those exact keys fall through to `return raw, []` (`deepread_sdk/enrich.py:86`), which would store blob-like output as prose TL;DR and degrade downstream QA display/retrieval metadata. The guard should treat any JSON/fence-looking response as non-prose unless it is successfully parsed or leniently extracted.

- Section offset metadata does not round-trip to the stored `content`. `recover_structure` strips section content but stores unstripped/or heading-inclusive offsets: no-heading documents store `content=text.strip()` with `start_pos=0, end_pos=len(text)` (`deepread_sdk/structure.py:57-61`); titled single-section documents store stripped content from `title_line_end` to EOF (`deepread_sdk/structure.py:67-71`); normal sections store `content=text[content_start:end].strip()` but `start_pos=pos`, i.e. the heading line position, not the content start (`deepread_sdk/structure.py:83-88`). `Reader.json` exposes these as canonical `{content,start_pos,end_pos}` (`deepread_sdk/reader.py:76-84`), so consumers cannot reliably use offsets to recover the exact content slice from `raw_md`. This can corrupt retrieval/index provenance. Either make `content == raw_md[start_pos:end_pos]` or clearly adjust offsets to match the stripped content span.

- The token fallback can return `0` for non-empty strings. When `tiktoken` is unavailable or encoding fails, `count_tokens` returns `len(text) // 4` (`deepread_sdk/tokens.py:23`), so `"a"`, `"ab"`, and `"abc"` report zero tokens. The contract explicitly requires `count_tokens("") == 0`; non-empty content should not normally have zero token count because these counts feed section inventory and budgeting. Use `max(1, math.ceil(len(text) / 4))` or equivalent for non-empty fallback text.

- `process_one` trusts the enricher to return exactly one section TL;DR per section and indexes blindly (`deepread_sdk/build.py:24-29`). The bundled `Enricher` currently satisfies this, but the module contract says `process_one` builds a full `DocRecord`, and build isolation should prevent enrichment oddities from crashing single documents. A shorter/malformed custom enricher result causes `IndexError`, making the doc fail instead of using deterministic fallbacks. Normalize/pad/truncate `sec_tldrs` in `process_one` or validate and fallback per section.

VERDICT: CHANGES REQUESTED

1. Harden `parse_global_response` so any JSON/fenced/blob-looking response that cannot be parsed/extracted does **not** become the TL;DR; return `("", keywords)`/`("", [])` instead.  
2. Fix section `content`/`start_pos`/`end_pos` consistency so `Reader.json` offsets correspond exactly to the exposed content, or preserve unstripped raw slices consistently.  
3. Ensure the token-count fallback returns zero only for empty input and at least one token for non-empty input.  
4. Make `process_one` robust to incorrect `section_tldrs` length by padding/truncating/falling back instead of raising.
